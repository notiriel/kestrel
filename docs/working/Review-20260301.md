# Architectural Review — 2026-03-01

Evaluation of the Kestrel codebase against hexagonal architecture, DRY, and SOLID principles.

**Codebase size**: ~11k lines source (domain: 2.0k, adapters: 6.3k, ports: 165, UI components: 2.0k, extension.ts: 676), ~4.9k lines tests.

---

## 1. Hexagonal Architecture

The hexagonal (ports-and-adapters) boundary is the strongest aspect of this codebase. The domain core is genuinely pure — zero `gi://` imports across all 14 domain files, fully testable with Vitest, and all dependencies flow inward.

### What works well

- **Clean domain boundary**: No domain file imports from `ports/`, `adapters/`, or `ui-components/`. Verified by inspection and partially by `test/arch/domain-boundary.test.ts`.
- **Port abstractions**: 11 port interfaces define the adapter contracts. Most are well-scoped (e.g., `MonitorPort` has just 2 methods, `ConflictDetectorPort` has 2).
- **WorldUpdate as universal return type**: All domain operations return `{ world, scene }`, giving adapters everything they need to render without computing layout themselves.
- **Scene model**: `computeScene()` produces a complete physical-state snapshot, keeping rendering logic testable without GNOME. This is textbook hexagonal — the domain computes the output model, adapters just apply it.
- **No barrel files**: Every import is a direct path. No index.ts re-export indirection.

### Boundary violations

| Violation | Severity | Location |
|-----------|----------|----------|
| `KeybindingPort` imports `gi://Gio` | **High** | `src/ports/keybinding-port.ts:1` |
| `question-card.ts` imports 5 runtime domain functions | **High** | `src/ui-components/question-card.ts:3-12` |
| `HelpOverlayAdapter` lives in `ui-components/` but is a full adapter | **Medium** | `src/ui-components/help-overlay.ts` |
| `extension.ts` downcasts `ClonePort`→`CloneAdapter` for diagnostics | **Medium** | `src/extension.ts` (3 locations) |
| `extension.ts` imports `Meta` from `gi://Meta` | **Low** | `src/extension.ts:40` |
| `PanelIndicatorPort.update()` takes full `World` aggregate | **Low** | `src/ports/panel-indicator-port.ts` |

**`keybinding-port.ts`**: Ports must not import platform types. The `Gio.Settings` parameter should be abstracted (e.g., `settings: unknown` with the adapter performing the cast).

**`question-card.ts`**: The most significant violation. CLAUDE.md states UI components "must NOT import domain types or adapter state," yet this file imports `parseQuestion`, `selectQuestionOption`, `setOtherText`, `shouldAutoAdvance`, and `formatQuestionResponse` from `domain/notification.js` — runtime value imports, not just types. A UI component is directly executing domain logic and managing its own state machine. This bypasses the adapter/controller layer entirely. The adapter-UI boundary test (`test/arch/adapter-ui-boundary.test.ts`) does not scan `ui-components/` for domain imports, so this goes undetected.

**`HelpOverlayAdapter`**: Manages `Main.layoutManager`, handles keyboard events, manages visibility state — this is adapter behavior, not presentational. Should live in `src/adapters/`.

### Dependency inversion in `extension.ts`

The composition root correctly types most fields by their port interfaces:

```typescript
private _monitorAdapter: MonitorPort | null = null;
private _windowAdapter: WindowPort | null = null;
private _focusAdapter: FocusPort | null = null;
// ... 6 more port-typed fields
```

However, several fields use concrete types with no port abstraction:

```
OverviewHandler, NavigationHandler, WindowLifecycleHandler,
SettlementRetry, NotificationCoordinator, HelpOverlayAdapter,
MouseInputAdapter, PanelIndicatorAdapter (port exists but unused)
```

The handlers are orchestrators that arguably don't need port abstractions, but `PanelIndicatorAdapter` having an unused port interface is a direct DIP violation.

## 2. DRY (Don't Repeat Yourself)

### Domain duplication

**Slot-targeting fallback** — The pattern `windowAtSlot(targetWs, sourceSlot) ?? lastWindow` appears in three independent implementations:
- `navigation.ts` `focusVertical()`
- `window-operations.ts` `moveVertical()`
- `world.ts` `navigateFromEmptyWorkspace()`

All implement "find the window at the same horizontal slot in the target workspace, fall back to last window." This should be a shared helper in `workspace.ts`.

**`replaceWorkspaceAt` pattern** — `workspaces.map((w, i) => i === idx ? newWs : w)` appears 4+ times in `world.ts` (`removeWindow`, `setWindowFullscreen`, `widenWindow`, `moveVertical`). A one-line helper would eliminate this.

**`pruneEmptyWorkspaces` + `ensureTrailingEmpty` pair** — Always called together (end of `removeWindow`, end of `moveVertical`, inside `restoreWorld`). No convenience function wraps the pair.

**Notification state Map copy** — `const notifications = new Map(state.notifications); notifications.set(id, ...); return { ...state, notifications }` repeats ~8 times in `notification.ts`. A `updateNotification(state, id, updater)` helper would halve this boilerplate.

### Adapter duplication

**Status priority logic** — Identical `needs-input: 3, working: 2, done: 1` priority ranking duplicated in:
- `panel-indicator-adapter.ts` `_highestPriority()`
- `extension.ts` `_getWorkspaceClaudeStatus()`

This is domain logic that belongs in one place (e.g., `notification.ts` or `notification-types.ts`).

**`Easeable` interface** — The workaround `interface Easeable { ease(params: Record<string, unknown>): void }` is duplicated in 3+ files: `clone-adapter.ts`, `notification-overlay-adapter.ts`, `notification-focus-mode.ts`. Should be defined once in `animation-helpers.ts` (which already exists and defines `easeOrSet`).

**`OVERVIEW_LABEL_WIDTH = 56`** — Hardcoded in `clone-adapter.ts` and appears as magic number `56` in two places in `overview-handler.ts`. Should be a shared constant.

### Test duplication (most significant DRY issue)

**No shared test fixtures exist.** Every test file is fully self-contained, leading to:

| Duplicate | Count | Files |
|-----------|-------|-------|
| `wid()` helper (`\`win-${n}\` as WindowId`) | 12 | Every test file |
| `config`/`monitor` fixture literals | 9+ | Every domain + adapter test |
| `makeWorld()` factory | 3 | navigation, window-ops, overview tests |
| `makeMultiWorld()` factory | 3 | world, navigation, window-ops tests |
| `buildWorld()` factory | 2 | navigation-handler, overview-handler tests |
| `mockMetaWindow()` factory | 2 | extension, window-lifecycle-handler tests |

A `test/helpers/fixtures.ts` module exporting `wid()`, `defaultConfig`, `defaultMonitor`, `makeWorld()`, and `makeMultiWorld()` would eliminate ~200 lines of duplication and ensure consistent test fixtures.

Additionally, `mock-ports.ts` only exports mocks for 2 of 11 ports. The remaining 9 are mocked inline in individual test files, sometimes duplicating each other.

## 3. SOLID Principles

### S — Single Responsibility

**`world.ts` (470 lines, 20+ exported functions)** — God module. Mixes:
1. Aggregate root type definition
2. World creation and restoration
3. Window lifecycle (add, remove, fullscreen, widen)
4. Focus management
5. Workspace management (switch, rename, prune, ensure trailing)
6. Search/filtering
7. Config/monitor updates
8. Notification plumbing

Suggested split: `world-core.ts` (type + create + utils), `window-lifecycle.ts` (add/remove/fullscreen), `workspace-management.ts` (switch/prune/ensure/rename/filter).

**`notification.ts` (489 lines, 26+ exported functions)** — Second God module. Bundles:
1. Notification lifecycle (add, respond, dismiss)
2. Session/window registration and status tracking
3. Question interaction state machine (navigate, select, format response)
4. Focus mode state machine (enter, exit, navigate, sync)
5. Payload classification (`classifyPermissionPayload` — inspects raw DBus payload shapes)
6. Response parsing

Items 3 and 4 are complex enough for their own modules. Item 5 is anti-corruption layer logic that arguably belongs closer to the adapter boundary.

**`CloneAdapter` (~920 lines)** — Largest adapter. Handles clone lifecycle, visual allocation, workspace containers, scene rendering, overview rendering, filter/rename UI, focus indicator, and diagnostics. The rename UI (keyboard handling for `St.Entry`, ~100 lines) is particularly out of place — input behavior unrelated to clone layout.

**`overview-handler.ts` (~625 lines)** — Handles overview enter/exit, hit testing, drag-to-reorder, filter input, rename orchestration, and overview transform computation. The `_computeTransform` / `_computeWorkspaceWidth` methods are pure layout math that belongs in the domain. `_navigateFilteredVertical` is navigation policy that belongs in the domain's overview module.

**Clean SRP modules** (positive examples):
- `fuzzy-match.ts` — single function
- `viewport.ts`, `window.ts` — pure data + factory
- `navigation.ts` — exclusively focus movement
- `layout.ts` — exclusively pixel math
- `NavigationHandler` — 3 methods, clean guard→domain→apply pattern

### O — Open/Closed

The port interface design enables extension through new adapter implementations without modifying existing code. The `ClonePort` even defines narrow sub-interfaces (`CloneLifecyclePort`, `CloneRenderPort`, `OverviewRenderPort`, `OverviewFilterPort`) for this purpose — though consumers currently use only the wide `ClonePort` composite, negating the benefit.

`KestrelConfig` is open to new fields without changing consuming code, since consumers destructure only what they need.

No plugin/strategy pattern issues noted — the codebase doesn't need runtime extensibility.

### L — Liskov Substitution

Not broadly applicable (few class hierarchies). The port/adapter relationship is the main substitutability concern, and it works: mock implementations in tests substitute for real adapters through port interfaces.

One concern: `WindowPort.setWorkAreaY()`/`setMonitorBounds()` are imperative configuration methods called after construction. If a substitute implementation required different initialization order, the implicit contract would break. Constructor injection would be safer.

### I — Interface Segregation

**Well-segregated**: `MonitorPort` (2 methods), `ConflictDetectorPort` (2 methods), `ShellPort` (3 methods), `NotificationPort` (4 methods), `WindowEventPort` (callbacks struct pattern).

**Candidates for splitting**:

| Port | Current Methods | Suggested Split |
|------|----------------|-----------------|
| `FocusPort` | 8 methods (track, untrack, getMetaWindow, focus, focusInternal, connectFocusChanged, openNewWindow, closeWindow) | `FocusRegistryPort` (track/untrack/get) + `FocusActivationPort` (focus/connect) + `WindowCommandPort` (open/close) |
| `ClonePort` | Wide composite of 4 sub-interfaces | Already split but consumers don't use narrow types |

`FocusPort` is the clearest ISP violation: `focus` vs `focusInternal` is an implementation detail leaking through the port boundary, and `openNewWindow`/`closeWindow` are unrelated to focus management.

### D — Dependency Inversion

Mostly well-applied. High-level modules (extension, handlers) depend on port abstractions. The domain depends on nothing external.

Violations noted above: 8 fields in `extension.ts` use concrete types, `PanelIndicatorAdapter` typed concretely despite having a port, diagnostics code downcasts past port interfaces.

## 4. Additional Observations

### Test coverage gaps

- **20+ adapter files have zero test coverage** — all actual GNOME adapters are untested. Only the 3 handler orchestrators and extension.ts have tests.
- **Architecture boundary tests are incomplete**: They guard `extension.ts`→domain and `adapters/`→UI, but not `ui-components/`→domain (which is currently violated) or `ports/`→GNOME (also currently violated).
- **Textual model notation** defined in CLAUDE.md is used in ~1 test comment out of ~100 spatial state descriptions. Most tests use ad hoc prose notation.

### Domain logic in wrong layers

Several pieces of domain-level logic live in adapter/handler code:

| Logic | Location | Should be in |
|-------|----------|-------------|
| Maximize → 2-slot policy | `window-lifecycle-handler.ts:79` | `domain/world.ts` `addWindow` |
| Overview transform computation | `overview-handler.ts:580-615` | `domain/layout.ts` or `domain/overview-state.ts` |
| Workspace width calculation | `overview-handler.ts:608-615` | `domain/layout.ts` |
| Filtered workspace navigation | `overview-handler.ts:121-134` | `domain/overview.ts` |
| `overviewActive: true` manual set | `overview-handler.ts:150` | Domain function |
| Status priority ranking | `panel-indicator-adapter.ts` + `extension.ts` | `domain/notification-types.ts` |
| `_listWorkspaces` filtering | `extension.ts` | Domain function |
| `_visitSession` domain query | `extension.ts` | Domain function |
| `_getWorkspaceClaudeStatus` | `extension.ts` | Domain function |

### Signal handling inconsistency

Signal ID initialization is inconsistent: some adapters use `number | null` (check `!== null`), others use `number` initialized to `0` (check truthiness). Both patterns work but the inconsistency makes the codebase harder to scan.

### Composition root complexity

`extension.ts` at 676 lines contains non-trivial orchestration logic beyond wiring. Methods like `_applyUpdateWithScroll`, `_handleExternalFocus`, `_visitSession`, `_listWorkspaces`, and `_getWorkspaceClaudeStatus` contain domain coordination and business logic. The `NotificationCoordinator` constructor receives a 9-callback options object with extensive `this` capture — fragile and hard to follow.

## 5. Summary Scorecard

| Principle | Grade | Notes |
|-----------|-------|-------|
| **Hexagonal boundary** | **A-** | Clean domain isolation; 2 high-severity boundary violations (keybinding port, question card) |
| **DRY — Domain** | **B** | Several repeated patterns (slot targeting, workspace replace, prune+ensure) |
| **DRY — Adapters** | **B+** | Status priority and Easeable interface duplicated; mostly clean |
| **DRY — Tests** | **C** | No shared fixtures; wid/config/monitor/makeWorld duplicated across 9-12 files |
| **Single Responsibility** | **B-** | Two God modules (world.ts, notification.ts); CloneAdapter and overview-handler overloaded |
| **Open/Closed** | **A** | Port design enables extension; narrow sub-interfaces exist |
| **Liskov Substitution** | **A** | Mocks substitute cleanly through ports |
| **Interface Segregation** | **B+** | Most ports well-scoped; FocusPort too wide; ClonePort narrow split unused |
| **Dependency Inversion** | **B+** | Mostly port-typed; 8 concrete-typed fields in extension.ts |

### Top recommendations by impact

1. **Extract test fixtures** — Create `test/helpers/fixtures.ts` with shared `wid()`, config, monitor, and world factories. Highest effort-to-benefit ratio.
2. **Fix boundary violations** — Remove `gi://Gio` from `keybinding-port.ts`; extract domain function calls from `question-card.ts` into an adapter callback; move `help-overlay.ts` to `src/adapters/`.
3. **Split `world.ts`** — Extract window lifecycle and workspace management into focused modules. The file's 20+ exports signal it's doing too much.
4. **Move domain logic out of adapters** — Overview transform computation, status priority, and filtered navigation belong in the domain.
5. **Add arch boundary test for `ui-components/`** — The existing tests don't catch `question-card.ts` importing domain functions.
6. **Extract shared constants** — `OVERVIEW_LABEL_WIDTH`, `Easeable` interface, status priority ranking.
